## 1. Executive summary (<=200 words)

Portfolio_7's strict-`<` SQL predicate on `news_sentiment.published_at` is structurally safe for the article-level read path, but the surrounding plumbing has **three real look-ahead vectors** plus one schema-mismatch fragility:

1. **`market_data.avg_sentiment` aggregation refresh is NOT point-in-time.** `update_market_data_sentiment` (`NLP/persistence/repository.py:251-298`) does a *blanket* `UPDATE` of `market_data.sentiment_score` from `AVG(sentiment_score) GROUP BY DATE(published_at)` every time it runs. A 2020 row gets re-stamped today with FinBERT scores produced by today's checkpoint. The aggregate is a function of *current* article inventory and *current* model, not the inventory/model that existed at the time of the bar. **VERDICT: RISK** if any P7 caller uses the `market_data` fallback (A2's `SENTIMENT_FALLBACK_TO_MARKET_DATA=True` does).
2. **FinBERT checkpoint is unpinned and the configured local directory does not exist on disk.** `NLP/core/paths.py:15` points `MODEL_DIR` at `finbert-combined-final/`, but the disk has `finbert-finetuned-final/`. `_resolve_model_dir` (`NLP/sentiment/scorer.py:60-70`) silently falls back to remote `ProsusAI/finbert` — a mutable Hub repo. **VERDICT: VIOLATION** of A2 §9 risk-1 mitigation. Backtests run today will use a different model than backtests run in 6 months.
3. **`published_at` is the upstream API publish timestamp, *not* an ingestion timestamp.** There is no `ingested_at` column. Articles back-filled in 2026 for 2020 dates land with `published_at=2020-xx-xx` and `created_at=NOW()`, but P7's SQL filters on `published_at < context.time` only. If an article truly entered the wire on 2020-01-15 at 09:30 and a 2020-01-15 16:00 P7 bar consumes it, that's fine. But the system has **no defense** against late-arriving wire stories (e.g., FMP backfilling an article that wasn't online until much later). **VERDICT: RISK**, acceptable but must be documented and surfaced through `created_at >= published_at` invariant assertion in CI.
4. **Schema mismatch (B4 in A SYNTHESIS):** the schema declares `market_data.avg_sentiment` (`schemaDefinitions.py:51`) but writes go to `market_data.sentiment_score` (`repository.py:255`). P7 A2's `COALESCE` fallback is correct.

The article-level path (`_SENTIMENT_QUERY` in A2 §5.2) uses strict `<` and is **SAFE**. The risk surface is the *fallback* path and the *refresh semantics* of the daily aggregate.

We deliver: lookahead-vector taxonomy, per-vector verdict, a CI gate pytest under `tests/portfolios/test_p7_lookahead.py`, a model-pinning unified diff (proposed, not applied), and a strict published_at-only flag recommendation.

## 3. Audit findings (file:line evidence)

### 3.1 Schema (`news_sentiment`)

`NLP/persistence/repository.py:57-68` (canonical at runtime):

```
ticker            VARCHAR(10)
article_url       TEXT
published_at      TIMESTAMP             <-- upstream wire publish time
sentiment_score   FLOAT
content_summary   TEXT
created_at        TIMESTAMP DEFAULT NOW()   <-- when WE inserted the row
```

Two timestamps are present. **`published_at` is the FMP `publishedDate`** parsed at `NLP/scrapers/fmp.py:130-132` (`datetime.strptime(pd_str, "%Y-%m-%d %H:%M:%S")`). **`created_at` is the row insertion time.** There is no separate `ingested_at` (e.g., when our scraper *first observed* the URL upstream) — `created_at` doubles as that.

Implication: a backfill run today for a 2020 ticker inserts `published_at=2020-...`, `created_at=2026-05-20`. The pair `(created_at - published_at)` is the *staleness gap*. P7's SQL `WHERE published_at < context.time` filters only on `published_at` and is **structurally blind** to whether the article was actually online on `context.time`'s wall clock. Cf. source #9 — backtest article inventory can diverge from real-time.

### 3.2 `news_sentiment` write path (per-article)

- `NLP/persistence/repository.py:137-184` (`insert_sentiment_record`): single-row insert, checks `check_article_exists(article_url)` first → idempotent on `article_url`.
- `NLP/persistence/repository.py:186-249` (`process_articles_with_sentiment`): bulk insert via `bulk_inject_to_db(..., conflict_columns=["article_url"])`. `MQSDBConnector.bulk_inject_to_db` (`src/common/database/MQSDBConnector.py:151-157`) injects `ON CONFLICT (article_url) DO NOTHING`. Per PostgreSQL semantics (source #12), this **never updates** an existing row.
- Therefore: **rerunning FinBERT on the same article URL produces a new in-memory score, but the DB row stays at the original score.** The historical `sentiment_score` in `news_sentiment` is sticky once written.

Caveat: if FinBERT is rerun on an article that was never inserted because of a transient failure, the *new* model version will score it. This is a small but real cross-model contamination in the per-article store, bounded by the gap rate.

### 3.3 Aggregate write path (`market_data` daily mean)

`NLP/persistence/repository.py:251-298` (`update_market_data_sentiment`):

```sql
UPDATE market_data md
SET sentiment_score = ns.avg_score
FROM (
    SELECT ticker, DATE(published_at) AS day, AVG(sentiment_score) AS avg_score
    FROM news_sentiment
    WHERE ticker = %s
    GROUP BY ticker, DATE(published_at)
) ns
WHERE md.ticker = ns.ticker
  AND md.date = ns.day
  AND (md.sentiment_score IS NULL OR md.sentiment_score != ns.avg_score);
```

This **rewrites the daily mean across all historical dates every time it runs**, with no PIT guard. Two failure modes:

(a) **Article-set drift.** Suppose on 2020-06-15 there were 2 articles for AAPL (scores +0.3, -0.1, mean +0.1). A 2026-05-20 backfill discovers a third 2020-06-15 article (score +0.9). The next `update_market_data_sentiment` re-stamps `market_data.sentiment_score` for AAPL/2020-06-15 to `mean(0.3, -0.1, 0.9) = 0.367`. Any backtest of P7 with `SENTIMENT_FALLBACK_TO_MARKET_DATA=True` (A2 default!) and `context.time = 2020-06-16` now reads a value that **could not have been computed on 2020-06-16**.

(b) **Model drift.** Even if the article set were frozen, if FinBERT is re-downloaded and the scorer re-runs on articles that were *not yet in `news_sentiment`* (e.g., previously failed inserts), the average changes — and gets propagated by the next aggregate-update — into a 2020 row.

Mitigations the code already has:
- The per-article `news_sentiment.sentiment_score` is sticky (3.2 above). Pure rescoring without new articles does **not** change the aggregate.
- The `WHERE` clause `md.sentiment_score IS NULL OR md.sentiment_score != ns.avg_score` short-circuits no-op updates.

Neither mitigation prevents (a). (a) is the dominant risk vector.

### 3.4 P7 reader path (A2 §5.2)

`_SENTIMENT_QUERY` (A2 §5.2 line 232-239):

```sql
SELECT ticker, published_at, sentiment_score
FROM news_sentiment
WHERE ticker = ANY(%s::text[])
  AND published_at >= %s
  AND published_at <  %s
  AND sentiment_score IS NOT NULL
```

Bound by `_fetch_sentiment_ex_ante` with `start_ts = cutoff_ts - 2 * SENTIMENT_WINDOW_DAYS` and `cutoff_ts = pd.Timestamp(context.time)`. **Strict `<` on the upper bound** — this is the right discipline (source #6, #7, #8, #10). VERDICT for this path: SAFE.

Fallback (`_MARKET_DATA_SENTIMENT_QUERY`, A2 §5.2 line 242-250):

```sql
SELECT ticker, date, COALESCE(avg_sentiment, sentiment_score)::float8 AS s
FROM market_data
WHERE ticker = ANY(%s::text[])
  AND date >= %s
  AND date <  %s
  AND COALESCE(avg_sentiment, sentiment_score) IS NOT NULL
```

Bound with `cutoff_date = cutoff_ts.date()`. Strict `<` on date — same hygiene. BUT: the *value* of `s` for a row with `date=2020-06-15` could have been computed by an aggregate-refresh on 2026. **VERDICT: RISK** — the SQL is PIT in *date*, but the *value* it returns is not PIT in *content*.

### 3.5 FinBERT model resolution

`NLP/core/paths.py:15`:
```
MODEL_DIR: Path = NLP_DIR / "finbert-combined-final"
```

Actual disk contents (`ls NLP/`):
```
finbert-finetuned-final/   model.safetensors (439 MB)
```

`NLP/sentiment/scorer.py:60-70`:
```python

    return explicit

    return DEFAULT_LOCAL_MODEL_DIR
```

Because `MODEL_DIR` *path-name does not match the dir on disk*, `_resolve_model_dir` will log a warning and silently load the **remote, unpinned, mutable** `ProsusAI/finbert`. **VERDICT: VIOLATION** — directly contradicts A2 §9 risk-1 mitigation that calls for pinning to the local safetensors copy.

### 3.6 Scrapers and timestamps

- FMP scraper (`NLP/scrapers/fmp.py:130-150`): parses `publishedDate` from FMP's JSON. Articles outside `[user_start, user_end]` are dropped. No `created_at`/`ingested_at` capture at scrape time — the DB's `DEFAULT NOW()` is the only ingestion-time signal.
- Alpha Vantage scraper (`NLP/scrapers/alpha_vantage.py:52-64`): parses `time_published` via `normalize_timestamp`. **`normalize_timestamp` (`NLP/core/timestamps.py:21-37`) returns a tz-naive Timestamp** by calling `.tz_convert(None)`. This drops any timezone information. Downstream code in P7 (`_fetch_sentiment_ex_ante`) calls `cutoff_ts.tz_localize("UTC")` when `tzinfo is None` — which means a UTC tz is *assumed* over what may have been ET, JST, etc. **Minor RISK**, a few hours of misalignment on cross-region wires.
- Yahoo scraper (`NLP/scrapers/yahoo.py:60-63`): `normalize_timestamp(content.get("pubDate"))` — same tz-strip behaviour.
- Finviz scraper (`NLP/scrapers/finviz.py:102`): `normalize_timestamp(row.get("Date"))` — Finviz timestamps are typically already date-only / day granularity; no minute precision.
- Aggregator (`NLP/scrapers/aggregator.py:104-142`): `_merge` dedups on `(publishedDate, title)`. A re-scrape with a corrected timestamp would *not* update an existing row (drop_duplicates keep="first"), so per-article CSV is also sticky.

## 4. Lookahead-vector taxonomy

| # | Vector | Mechanism | Vulnerable code | Triggered when |
|---|---|---|---|---|
| V1 | **Publication-time leakage (per-article)** | Article's `published_at` is the wire publish time; a same-bar article (16:00 ET wire vs 16:00 ET bar) could leak | `_fetch_sentiment_ex_ante` `_SENTIMENT_QUERY` | If predicate were `<=` instead of `<` |
| V2 | **Ingestion-time leakage** | An article published 2020-01-15 may not have been *online* until much later; backtest gets it via `published_at` | `news_sentiment.published_at` column + reader SQL | Always (architectural — no `ingested_at`) |
| V3 | **Model-checkpoint drift** | Re-downloading FinBERT in 2026 produces different scores than 2020 FinBERT would have | `_resolve_model_dir` HF fallback; absent revision pin | Every fresh container / pod / venv |
| V4 | **Aggregation refresh** | `update_market_data_sentiment` re-stamps historical `market_data.sentiment_score` based on *current* article inventory | `repository.py:251-298` UPDATE | Every NLP daemon cycle, retroactively to all history |
| V5 | **Schema mismatch** | Schema declares `avg_sentiment`, code writes `sentiment_score` — reader may silently read NULL | `schemaDefinitions.py:51` vs `repository.py:255` | If P7 reader hits `avg_sentiment` and code writes only `sentiment_score`, NULLs propagate |
| V6 | **Timezone strip** | `normalize_timestamp` returns tz-naive; downstream assumes UTC | `NLP/core/timestamps.py:34-37` | Non-UTC sources (Alpha Vantage time_published is UTC; Yahoo is UTC; but assumption is fragile) |

## 5. Verdict per vector

| # | Vector | Verdict | Rationale |
|---|---|---|---|
| V1 | Publication-time leakage | **SAFE** | A2's `_SENTIMENT_QUERY` uses strict `<` (A2 §5.2 line 237). Verified in P7's test sketch (A2 §7 `test_strict_less_than_cutoff`). The audit test below regression-locks this. |
| V2 | Ingestion-time leakage | **RISK** | No `ingested_at` column; pipeline trusts `published_at` from the upstream vendor. Magnitude is bounded by typical wire latency (seconds-to-minutes); acceptable for monthly-rebalanced P7, but must be documented. Mitigation: add `ingested_at` column **OR** add invariant `published_at >= created_at - INTERVAL '7 days'` to flag suspicious backfills. |
| V3 | Model-checkpoint drift | **VIOLATION** | `MODEL_DIR` typo (`finbert-combined-final` vs on-disk `finbert-finetuned-final`) forces HF Hub fallback to unpinned `ProsusAI/finbert`. Sources #4, #5, #13 confirm Hub is mutable. Direct contradiction of A2 §9 risk-1. |
| V4 | Aggregation refresh | **RISK** (severe if fallback enabled) | `update_market_data_sentiment` (`repository.py:251-298`) blanket-updates all historical `market_data.sentiment_score` on every NLP cycle. ## 8. Falsification

**Concrete falsification protocol** (apply-ready, can be cron'd into CI):

1. **Hermetic CI gate (always-on):** run `pytest tests/portfolios/test_p7_lookahead.py` on every PR. Tests `test_v1_strict_less_than_in_sql`, `test_no_row_at_or_after_cutoff`, `test_v3_finbert_local_checkpoint_exists`, and `test_v4_market_data_fallback_disabled_by_default` are pure-Python and require no DB; they MUST pass before merge.

2. **Live-DB falsification (weekly cron):** set `MQS_AUDIT_LIVE_DB=1` and run `test_live_db_no_post_cutoff_rows` against the production `news_sentiment` table with `cutoff = 2024-01-01`. If any row leaks: audit fails, alert pager.

3. **Restatement detector (V4 monitor):** snapshot `market_data.sentiment_score` for a representative set of (ticker, date) pairs once per week. Diff against last week's snapshot. Any change to a row with `date < (today - 30d)` is a V4 incident — it means the aggregate refresh re-stamped a historical row. Log to a dedicated audit table.

4. **Checkpoint hash invariant (V3 monitor):** at scorer initialization, compute and log `sha256` of `model.safetensors`. Any change between backtest runs is a V3 incident. Fail closed on `MQS_REPRODUCIBLE=1` env.

5. **OOS falsification on P7 itself:** if the canary tests above pass but P7's OOS Sharpe is *zero* (not negative — zero), that's also suspicious: it could mean the tilt is being silently neutralized by a bug. P7 SHOULD have *some* nonzero signal; if it's pure noise, look for V2/V4 contamination in the *fallback* path.

## 9. Risks and rollback path

### Risks of the proposed fix

| Risk | Impact | Mitigation |
|---|---|---|
| Renaming `MODEL_DIR` in `paths.py` breaks any script that imports `MODEL_DIR` and expects the typo'd path | Very low — the on-disk dir is the *correct* one; only the constant is wrong | Search confirms no callers expect `finbert-combined-final` literal; the only consumer is `_resolve_model_dir`, which checks `os.path.exists` |
| Disabling `SENTIMENT_FALLBACK_TO_MARKET_DATA` reduces P7's signal coverage on tickers with few `news_sentiment` rows | Some tickers will get `z=0` (no tilt) instead of a fallback value | Acceptable: A2 §4.2 already gates the fallback by `MIN_ARTICLES_PER_TICKER` and assigns `z=0` to under-supported names; the change here just makes that more conservative |
| Hard-failing on missing local FinBERT may break ad-hoc scripts in dev environments | Devs without the 439 MB checkpoint can't run NLP locally | Gate behind `MQS_ALLOW_HUB_FALLBACK=1` env var (proposed diff above); CI and prod set `MQS_REPRODUCIBLE=1` and require local checkpoint |
| The audit test `test_v3_finbert_local_checkpoint_exists` will fail until the `paths.py` typo is fixed | This is the desired behavior — the test exposes the bug | Apply the one-line `paths.py` diff at the same time as enabling the test |

### Rollback path

- **paths.py diff:** trivially reversible (`git revert`). The on-disk dir is unchanged.
- **scorer.py hardening:** opt-in via env var; default behavior preserved when `MQS_ALLOW_HUB_FALLBACK=1`.
- **P7 config fallback default:** flip back to `true` in the JSON config; no code change required.
- **None of the changes touch DB schema or DB data.** All rollbacks are config + code only.
